# 실시간 러닝 기록 기능 검토 요약

## 목적
- RunOn 앱에 스트라바/나이키런클럽(NRC)과 유사한 "직접 달리면서 기록" 기능을 도입할 수 있는지 판단
- 구현 복잡도를 MVP와 고도화 단계로 분리해 현실적으로 정리

## 1) 구현 가능 여부
- 결론: **구현 가능**
- 현재 코드베이스에 이미 있는 기반
  - `services/appleFitnessService.js` 기반 HealthKit 연동 및 러닝 기록 조회 로직
  - `expo-location` 사용 기반(위치 권한/위치 사용)
  - iOS 권한/연동 설정(Info.plist, Pod, HealthKit 연동 가이드)

## 2) 복잡도 판단
- **MVP (기본 실시간 기록)**: 중간~높음
- **스트라바/NRC 수준 완성도**: 높음~매우 높음

복잡도가 올라가는 핵심 이유:
- 앱 백그라운드/잠금화면/권한 변화 상황에서 추적 지속성 보장
- GPS 오차/튀는 값 필터링, 페이스/거리 신뢰도 확보
- 다양한 디바이스/OS 환경에서 안정성 확보
- 배터리 소모 최적화

## 3) 구현 범위 제안 (일정 제외)

### 3-1. 1차 범위 (MVP)
- 러닝 시작/일시정지/재개/종료
- 실시간 지표 표시 (시간, 거리, 페이스)
- 지도 경로(폴리라인) 표시
- 종료 후 결과 화면
- HealthKit 저장(쓰기 권한 포함)

### 3-2. 2차 범위 (고도화)
- 백그라운드/잠금화면 추적 안정화
- GPS 이상치 제거 및 정확도 개선
- 권한 거부/재요청/재진입 등 예외 UX 정교화
- 배터리 최적화
- 앱 중단 상황에서 데이터 유실 최소화

## 4) 결론
- 기능 자체 구현은 충분히 가능하다.
- 난이도를 결정하는 핵심은 실시간 화면 구현보다, **운영 환경에서의 안정성/정확도/예외처리 완성도**다.

## 6) 현재 논의 중: 러닝 시작 진입 위치
- 사용자 선호: `지도탭(MapTab)`에서 러닝 기록 시작
- 검토 결론(초안):
  - 지도탭 시작은 사용자 맥락(위치/코스 확인 후 시작)과 잘 맞아 우선 후보로 적합
  - 다만 지도탭은 이미 상단 검색/토글, 하단 BottomSheet가 있어 CTA 배치 충돌을 피해야 함
- 권장 UI 방향(초안):
  - 지도탭 우하단에 `러닝 시작` 플로팅 버튼(FAB) 배치
  - 탭 시 `시작 전 확인 시트` 노출(권한/GPS 상태/간단 안내)
  - 확인 후 전용 `RunningTrackerScreen`으로 진입(전체화면)
  - 종료 후 결과 화면을 거쳐 `MapTab`으로 복귀
