# Running Mini Map Implementation History

## 목적
- 러닝피드 카드와 러닝 종료 확인 모달에서 미니맵을 표시하고자 함.
- 요구사항: 카카오맵 타일 위에 실제 달린 경로(routeCoordinates)를 표시.

## 1차 구현: `RouteMap`을 카카오 WebView로 교체
- 변경:
  - 기존 SVG Polyline 기반 `RouteMap`을 카카오 SDK 로딩 WebView로 전환.
  - 시작/종료 마커, 경로선, bounds fit 추가.
- 기대:
  - 지도 타일 + 실제 경로를 동시에 표현.
- 실제 결과:
  - 로그상 `SDK_LOAD_OK`, `MAP_INIT_OK`, `ROUTE_RENDERED`가 찍히는 경우에도 UI는 검은 박스.

## 2차 진단: WebView 상태 코드 가시화
- 변경:
  - `MINIMAP_STATUS`, `MINIMAP_READY`, `MINIMAP_ERROR:*`를 RN으로 postMessage.
  - RN에서 로그 출력 및 오류 배지 노출.
- 확인된 사실:
  - 미니맵 렌더 로직(JS) 자체는 성공하는 케이스가 다수.
  - 실패는 로직보다 iOS WebView 렌더/합성 계층 가능성이 큼.

## 3차 안정화 시도: 자동 재로딩/프로세스 종료 대응
- 변경:
  - `onError`, `onContentProcessDidTerminate` 시 WebView 재생성 재시도.
  - map `relayout` + 재-bounds 적용.
- 기대:
  - 일시적인 WebView 프로세스 문제 완화.
- 실제 결과:
  - 일부 케이스에서 개선되었으나, 실기기에서 검은 화면 재발.

## 4차 구조 전환 시도: 단일 렌더러 + 캡처 캐시
- 목표:
  - 피드 카드마다 다수 WebView를 띄우지 않고 1개 렌더러로 순차 생성.
- 구현 A:
  - 숨겨진 WebView를 `react-native-view-shot`(`captureRef`)로 캡처하여 이미지 캐시.
- 구현 B:
  - `WebView.takeSnapshot()` 기반 캡처로 변경.
- 실제 결과:
  - 로그에서 `SNAPSHOT_UNSUPPORTED` 확인.
  - 오프스크린/숨김 렌더러의 `layoutWidth: 0` 상태도 함께 확인.
  - 결과적으로 프리뷰 이미지 생성 실패 -> "지도를 준비중입니다..." 무한 대기 발생.

## 5차 결론
- 현재 환경에서 "숨김 WebView 캡처 기반 프리뷰"는 신뢰성 부족.
- 문제 핵심:
  - 카카오맵 로직 자체보다, iOS WKWebView 렌더 합성/캡처 경로 안정성.

## 본 요청에 따른 조치 (당시 상태)
- 러닝피드/러닝 종료 모달의 미니맵 기능을 제거.
- 러닝 종료 모달에는 안내 문구만 남김:
  - "미니맵 표시는 일시적으로 비활성화되었습니다."

## 향후 재구현 시 권장 방향
- 옵션 A: 카카오맵 라이브 렌더 유지
  - 카드 단위 WebView 다중 생성 금지.
  - 가시 영역 제한(FlatList viewability 기반), 동시 인스턴스 최소화.
- 옵션 B: 서버/백엔드 정적 지도 이미지 파이프라인
  - 앱 단 WebView 캡처 없이 이미지 URL만 소비.
  - iOS 렌더/캡처 불안정성 회피 가능.

## 6차 재구현 (현재 적용)
- 적용 방향:
  - 캡처/큐/숨김 렌더러 구조를 완전히 제거.
  - `RouteMap`을 "라이브 카카오 WebView 렌더" 단일 경로로 단순화.
  - 피드 카드와 종료 모달에서 동일한 `RouteMap`만 사용.
- 안정화 포인트:
  - 로딩/오류 상태를 UI에서 즉시 확인 가능하도록 유지.
  - 오류 발생 시 카드 내부에서 즉시 재시도 가능.
  - 공유 카드(`RunningShareCard`)는 기존처럼 `provider="svg"`를 유지해 회귀 방지.
- 기대 효과:
  - 복잡도 감소로 디버깅/유지보수 용이.
  - 오프스크린 캡처 실패(`SNAPSHOT_UNSUPPORTED`) 경로 제거.

## 7차 추가 시도 (미해결 이력, 2026-05-04)
- 목표:
  - "피드에서 카카오맵이 아예 보이지 않는 문제"를 우선 해결.
  - 스크롤 중 경로 사라짐 현상 제거.

### 7-1) 피드 가시영역 기반 라이브 전환
- 변경:
  - 스크롤 오프셋으로 라이브 카카오 대상을 1~2개로 제한하고 나머지는 SVG 폴백.
- 결과:
  - 아래 항목에서는 SVG 경로가 보였지만, 스크롤 후 상단으로 이동 시 경로가 사라지는 사례 발생.
  - 동적 provider 전환이 오히려 상태 불안정(깜빡임/사라짐)을 유발.
- 판정:
  - **미해결** (피드 UX 불안정).

### 7-2) 강제 카카오 검증 더미 추가
- 변경:
  - `forceKakaoMiniMap` 플래그를 가진 테스트 더미 항목 추가.
  - 해당 항목은 가시영역 규칙/더미 SVG 규칙을 무시하고 항상 `provider="kakao"` 적용.
- 관찰 로그:
  - `SDK_LOAD_OK`, `MAP_INIT_OK`, `MINIMAP_READY(ROUTE_RENDERED)`까지 정상 수신.
- 결과:
  - 로그는 성공인데 화면에는 지도 미표시.
- 판정:
  - **미해결** (데이터/SDK 로직보다 렌더 합성 계층 문제 가능성 강화).

### 7-3) 강제 카카오 항목의 SVG 폴백 비활성화 + Ready 판정 완화
- 변경:
  - 강제 카카오 더미에는 SVG 폴백 금지(`disableSvgFallback`) 적용.
  - `MINIMAP_STATUS: MAP_INIT_OK` 시점도 ready로 인정.
- 결과:
  - "폴백 때문에 가려진 것"은 배제했지만, 여전히 피드에서 미표시.
- 판정:
  - **미해결**.

### 7-4) relayout/bounds 강제 반복 적용
- 변경:
  - `renderRoute` 후 `relayout + setBounds`를 지연 다회 호출.
  - `resize` 이벤트, RN `onLayout`에서도 강제 relayout 주입.
- 결과:
  - 체감상 표시 개선 없음.
- 판정:
  - **미해결**.

### 7-5) 레이아웃 안정화 롤백 (풀폭 음수 마진 제거)
- 변경:
  - 피드 미니맵 래퍼의 `marginHorizontal: -14` 제거.
  - 미니맵 폭 계산을 안정 폭(`screenWidth - 28`)으로 복귀.
- 결과:
  - 피드 카카오 미표시 현상 지속.
- 판정:
  - **미해결**.

## 7차 종합 결론
- 현재까지의 추가 시도에서 "피드 카카오 미니맵 미표시"는 재현 유지.
- 로그상 JS/Kakao 초기화와 경로 렌더 이벤트는 성공하므로, 문제축은 다음으로 좁혀짐:
  - iOS WKWebView의 리스트 셀 내 합성/표시 타이밍 문제
  - 피드 섹션 구조(스크롤/클리핑/레이아웃)와 WebView 결합 이슈
- 정리:
  - **해결되지 않은 시도들을 문서화 완료**.
  - 다음 단계는 피드 전용 최소 재현 셀(순수 WebView만 있는 카드)로 구조적 방해요소를 분리 진단하는 방식이 필요.
